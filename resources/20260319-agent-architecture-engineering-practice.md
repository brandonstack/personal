# Agent 原理、架构与工程实践

系统梳理 Agent 架构中影响工程效果的关键模块：控制流、上下文工程、工具设计、记忆、多 Agent 组织、评测、追踪和安全。以 OpenClaw 实现串联。

**核心发现**：更贵的模型提升没想象中大，Harness 和验证测试质量对成功率影响更大；调试 Agent 应优先检查工具定义；评测系统本身的问题比 Agent 问题更难发现。

## 1. Agent Loop

核心循环：感知 → 决策 → 行动 → 反馈，约 20 行代码。循环本身极稳定，新能力通过三种方式接入：扩展工具集、调整系统提示、状态外化到文件/DB。不应让循环变成状态机。

**Workflow vs Agent**：执行路径由代码写死 = Workflow，由 LLM 动态决定 = Agent。核心区别是控制权归属。

**五种控制模式**：
1. **提示链**：顺序步骤，适合线性流程（生成→翻译、大纲→正文）
2. **路由**：输入分类，定向专用流程（简单→轻量模型，复杂→强模型）
3. **并行**：分段法（拆子任务并发）/ 投票法（同任务多次取共识）
4. **编排器-工作者**：中央 LLM 动态分解委派，综合结果
5. **评估器-优化器**：生成→评估→循环，适合质量标准难精确定义的任务

## 2. Harness 比模型更关键

Harness = 验收基线 + 执行边界 + 反馈信号 + 回退手段。在高可验证任务（代码）上最成立，弱验证任务（研究、协商）中模型上限仍更关键。

**OpenAI 实践**（3 工程师 5 个月百万行代码，10x 速度）：
- Agent 看不到的等于不存在——知识存代码库本身，AGENTS.md ≤100 行做索引
- 约束编码化（Linter/CI/类型系统），不靠文档
- Agent 端到端自主（验证→修复→PR→合并），全链路无人介入
- 测试偶发失败用重跑，不阻塞进度

**任务象限**：目标清晰度 × 验证自动化程度。右上角（明确+自动验证）最适合 Agent。Harness 的目标是把任务推向右上角。

## 3. 上下文工程

注意力 O(n²)，上下文越长关键信号越被稀释。核心失效模式：无关内容占大头→决策质量下滑（Context Rot）。

**分层管理**：
- **常驻层**：身份、约定、禁止项——短、硬、可执行
- **按需加载**：Skills 描述符常驻，内容触发时注入
- **运行时注入**：时间、渠道 ID、用户偏好
- **记忆层**：MEMORY.md 跨会话，需要时读取
- **系统层**：Hooks/代码规则，不进上下文

> 确定性逻辑不进上下文，交给 Hooks/代码/工具。

**压缩策略**：滑动窗口（丢旧消息）、LLM 摘要（丢细节保决策）、工具结果替换（占位符替换）。压缩时保留优先级：架构决策 > 已修改文件 > 验证状态 > TODO > 工具输出。标识符（UUID/hash/URL）必须原样保留。

**Prompt Caching**：精确前缀匹配触发 KV 缓存。常驻层越稳定→命中率越高。反直觉：稳定的大系统提示比频繁变动的小提示成本更低（写入一次，读取 90% 折扣）。

**Skills 按需加载**：系统提示只保留索引（短描述符），完整内容触发时注入。描述符写法：路由条件而非功能介绍，"何时该用我"比"我能做什么"重要。加反例后准确率 53%→85%。Skill 数量要控制，低频不进默认列表。

**文件系统做上下文接口**（Dynamic Context Discovery）：工具结果写文件，Agent 按需 grep 读取。Cursor A/B 测试：MCP 工具 token 消耗减少 46.9%。压缩时把历史写文件保留可追溯性。

## 4. 工具设计

工具定义质量 > 数量。5 个 MCP 服务器≈55K tokens 工具定义开销。

**演进三代**：
1. API 封装——粒度过细
2. ACI（Agent-Computer Interface）——面向 Agent 目标，不是底层 API
3. Advanced Tool Use——动态发现（search_tools，准确率 49%→74%）、代码编排（token 150K→2K）、示例驱动（准确率 72%→90%）

**ACI 原则**：参数明确+结构化错误+修正建议+定义实现绑定（Zod schema）。调试先检查工具定义，多数选错源于描述不准确。

框架内部事件（压缩通知等）不进 LLM，只保留 user/assistant/tool_result 标准类型。

## 5. 记忆系统

四种记忆：
- **工作记忆**（上下文窗口）：当前任务最小信息
- **程序性记忆**（Skills）：操作流程，按需加载
- **情景记忆**（JSONL 历史）：发生了什么，支持检索
- **语义记忆**（MEMORY.md）：Agent 主动写入的稳定事实

**整合触发**：tokenUsage/maxTokens ≥ 0.5 时触发。成功→摘要追加 MEMORY.md，失败→原始消息归档。核心：只移动指针不删除原始消息，可回退。

大多数 Agent 不需要向量存储，结构化 Markdown + 关键词搜索够用，规模超几千条再考虑。

## 6. 自主度

**跨 session**：Initializer Agent 生成 feature-list.json + 进度文件，Coding Agent 循环执行，每次从文件恢复现场。进度放文件不放上下文，功能清单用 JSON 不用 Markdown。

**任务状态外化**：同时只有一个 in_progress，每完成一步先更新状态再继续。

**后台 I/O**：慢速 subprocess 放后台线程，通知队列注入结果，主循环不感知并发细节。

## 7. 多 Agent 组织

指挥者模式（同步紧密互动）vs 统筹者模式（异步委派，产出为 PR/分支等持久工件）。

组织方式：主 Agent 统筹 + 子 Agent 并行，JSONL inbox 通信，Worktree 隔离，任务图管理依赖。

**关键**：
- 子 Agent 只回传摘要，搜索/调试细节留自己上下文
- 协作方式写成协议（结构化消息，append-only，崩溃可恢复）
- 幻觉会互相放大→需交叉验证打断错误链
- 子 Agent 最小系统提示，不带 Skills/Memory，防权限外泄
- 顺序：协议先定→隔离先做→再谈协作并行

## 8. 评测

**结构**：task（测什么）→ trial（跑多少次）→ grader（怎么打分）。transcript（执行记录）+ outcome（环境最终状态）都要覆盖。

**指标**：Pass@k 验证能力边界（开发阶段），Pass^k 保证上线质量（回归测试），不能混用。

**评分器**（确定性从高到低）：代码评分器 > 模型评分器 > 人工评分器。有明确答案优先用代码评分。

**搭建**：20-50 个真实失败案例启动。环境隔离（每次干净状态）。正例+反例都覆盖。通过率接近 100% 时补更难的任务。

> **先修评测再改 Agent**——评测分数下降先查环境（资源不足/评分器 bug/测试脱节），再动 Agent。

## 9. 追踪

Trace 记录：完整 Prompt + messages[] + 工具调用 + 推理链 + 输出 + token/延迟。

**两层可观测性**：人工抽样标注（摸清失败模式）+ LLM 自动评估（全量覆盖），互相校准。

**采样策略**：负反馈 100% 入队列、高 token 消耗优先、固定时间段随机、模型变更后 48h 全量。

**事件流底座**：tool_start/tool_end/turn_end 三节点发事件，一次发布多路消费（日志/UI/评测/审查）。

## 10. OpenClaw 实现

五层架构：Gateway（WebSocket）→ Channel 适配器（23+ 渠道）→ Pi Agent（主循环）→ 工具集 → 上下文+记忆。

关键实现：MessageBus 解耦渠道与 Agent；系统提示按层叠加（SOUL.md + AGENTS.md + TOOLS.md + MEMORY.md + Skills）；cron/heartbeat 主动触发；任务进度写磁盘可恢复。

**安全边界先于功能**：白名单授权 → 工作空间路径隔离（execFile 防注入）→ 审计日志 → Prompt Injection 防护（标注不可信内容边界、敏感操作显式确认、关键路径独立 LLM 验证）→ Provider 故障切换。

## 落地顺序

1. 单渠道先跑通，不要第一版就抽象多渠道
2. 安全边界先于功能
3. 记忆整合要早做（第 20 轮后就会垮）
4. Skills 先于新工具
5. 第一个失败就建评测

## 八大反模式

1. 系统提示当知识库→约定留提示，知识移 Skills
2. 工具数量失控→合并重叠，明确命名空间
3. 验证闭环缺失→每类任务绑验收标准
4. 多 Agent 无边界→角色权限+worktree 隔离
5. 记忆不整合→超阈值自动触发
6. 没有评测→失败案例立刻转测试用例
7. 过早多 Agent→先验证单 Agent 上限
8. 约束靠期望→改用 Linter/Hook/工具验证
