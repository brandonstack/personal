## 本周回顾

### 完成了什么

- 启动了 **PKM Polish** 项目：确定"纯文本 + Git + Obsidian(UI) + Agent(自动化)"架构，完成 Obsidian 插件调研（Dataview, Templater, Tasks, Calendar, Periodic Notes 等），配好基础设施
- 启动了 **Work Wrap-up** 项目：开始梳理 Price Accuracy / Dashboard / pipelines 的交接文档，规划 6 月底 clean exit

- RSS/OPML 导出，为 feed dashboard 做准备

### 消化了什么

- **Harness Engineering 范式**：读了 Anthropic 博客的 generator-evaluator 架构（$9 vs $200 成本差但质量飞跃），以及 GoSailGlobal 的 Prompt→Context→Harness 三阶段演进梳理
- **AI Coding 的反思声音**：Jeremy Howard 批评 Claude Code 方向（AI 组合式创造力不能突破训练分布），Mario Zechner 警告 agentic coding 的错误复利和架构腐蚀
- **Vibe Coding 实战方法论**：Boris 的 Claude Code 高级技巧（远程控制、并行 worktree、自定义 agent），FradSer 的双阶段工作流，Superpowers 的五阶段结构化流程
- **MCP vs CLI 深度对比**：Justineo 的研究 — MCP 是外循环集成层，CLI 是内循环效率层，两者互补
- **职业转型思考**：离开大厂的真实体验（固定薪资舒适区、外部评价依赖、信息茧房三个陷阱），以及自己的裁员焦虑日记

### 学到了什么

- Harness 的核心不是多 agent 编排，而是 **design criteria 作为 steering mechanism** — 评估标准决定了生成质量的上限
- 快与慢的张力：vibe coding 追求速度，但 architecture/API 层必须手写，agent 适合在明确边界内执行
- 焦虑的底层模式：过度责任感 ← 童年完美主义 ← 安全感条件化，识别到这是一个持续运行的"程序"
- Obsidian Dataview 的 inline JS 表达式比 DQL 灵活得多，适合 Dashboard 动态内容

好的一面是，读了大量 AI coding 的正反观点后，自己的判断更清晰了：harness engineering 是真正的杠杆点，不是 vibe coding 那种"让 AI 全写"。这个认知可以直接转化成 career narrative — 我要讲的故事不是"我会用 AI 写代码"，而是"我知道怎么设计 AI 系统的质量闭环"。

焦虑日记写下来之后松了一口气。识别出"过度责任感"这个模式很重要，下周想继续用 reframing 练习，把它从自动反应变成可观察的信号。
