# 驾驭工程：在 Agent 优先的世界中利用 Codex

作者：Ryan Lopopolo，OpenAI 技术团队成员

## Summary

OpenAI 团队用 Codex 从零构建了一个内部产品，**0 行手写代码**，约 5 个月产出百万行代码、~1500 个 PR、3 名工程师起步。文章分享了全 Agent 开发模式下的系统性经验：知识组织（地图而非手册）、架构强制（不变量而非微管理）、可观测性接入（让 Agent 能自己 QA）、技术债的垃圾回收机制。核心理念：**人类指挥，Agent 执行**。工程师的角色从写代码转变为设计让 Agent 高效工作的环境。

---

## 从空仓库开始

2025 年 8 月末，第一个 commit 落入空仓库。

初始脚手架——仓库结构、CI 配置、格式化规则、包管理器设置、应用框架——全部由 Codex CLI + GPT-5 生成，参考一小组已有模板。连指导 Agent 如何在仓库中工作的 AGENTS.md 文件本身也是 Codex 写的。

没有任何预先存在的人类手写代码。从一开始，仓库就由 Agent 塑造。

5 个月后，仓库包含约百万行代码，涵盖应用逻辑、基础设施、工具链、文档和内部开发工具。期间约 1500 个 PR 被打开和合并，起初只有 3 名工程师驱动 Codex，平均每人每天 3.5 个 PR，团队扩展到 7 人后吞吐量反而**增加**了。这不是为产出而产出：产品已被数百名内部用户使用，包括每日活跃的重度用户。

在整个开发过程中，人类从未直接贡献任何代码。这成为团队的核心哲学：**不手写代码**。

## 重新定义工程师角色

不再亲手写代码，**引入了一种不同的工程工作方式，聚焦于系统、脚手架和杠杆**。

早期进展比预期慢——不是 Codex 能力不够，而是环境描述不充分。Agent 缺少向高层目标推进所需的工具、抽象和内部结构。工程团队的首要工作变成了**让 Agent 能做有用的事**。

实践中，这意味着深度优先工作：将大目标拆解为小构建块（设计、编码、review、测试等），提示 Agent 构建这些块，再用它们解锁更复杂的任务。当某件事失败时，修复方式几乎从来不是「再试一次」。因为推进的唯一方式是让 Codex 完成工作，人类工程师总是介入任务并问：「缺少什么能力？如何让它对 Agent 既可见又可执行？」

人类几乎完全通过 prompt 与系统交互：工程师描述一个任务，运行 agent，让它开一个 PR。为了将 PR 推进到完成，我们指示 Codex 在本地 review 自己的改动，在本地和云端请求其他特定 agent review，回应任何人类或 agent 的反馈，循环迭代直到所有 agent reviewer 满意（这实际上是一个 Ralph Wiggum Loop）。Codex 直接使用我们的标准开发工具（gh、本地脚本和仓库内嵌的 skills）来收集上下文，无需人类复制粘贴到 CLI。

人类可以 review PR，但不是必须的。随着时间推移，我们已将几乎所有 review 工作推向 agent-to-agent 处理。

## 提升应用可读性

随着代码吞吐量增加，瓶颈变成了人类 QA 能力。因为固定约束是人类的时间和注意力，我们致力于通过让应用 UI、日志和应用指标本身对 Codex 直接可读来增加 agent 的能力。

例如，我们让应用可以按 git worktree 启动，这样 Codex 每个改动都能启动和驱动一个独立实例。我们还将 Chrome DevTools Protocol 接入 agent 运行时，创建了操作 DOM 快照、截图和导航的 skills。这使 Codex 能直接复现 bug、验证修复、推理 UI 行为。

我们对可观测性工具做了同样的事。日志、指标和链路追踪通过本地可观测性栈暴露给 Codex，该栈对每个 worktree 是临时的（ephemeral）。Codex 在应用的完全隔离版本上工作——包括其日志和指标，任务完成后即拆除。Agent 可以用 LogQL 查询日志、用 PromQL 查询指标。有了这些上下文，像「确保服务启动在 800ms 内完成」或「这四个关键用户旅程中没有任何 span 超过两秒」这样的 prompt 就变得可行了。

我们经常看到单次 Codex 运行在一个任务上持续工作 6 小时以上（通常人类在睡觉时）。

## 仓库知识作为记录系统

上下文管理是让 agent 在大型复杂任务中有效工作的最大挑战之一。我们最早学到的教训很简单：**给 Codex 一张地图，而不是一本千页的指导手册**。

我们试过「一个大 AGENTS.md」的方式，以可预见的方式失败了：

- **上下文是稀缺资源。** 巨大的指令文件挤占了任务、代码和相关文档的空间——agent 要么漏掉关键约束，要么开始为错误的目标优化。
- **太多指导变成了「非指导」。** 当一切都是「重要的」，就没有什么是重要的。Agent 最终在局部做模式匹配，而非有意识地导航。
- **它立刻就腐烂了。** 单体手册变成陈旧规则的坟场。Agent 分不清什么还是真的，人类停止维护它，文件悄然变成一个有吸引力的麻烦。
- **难以验证。** 单一大块不适合机械检查（覆盖率、新鲜度、所有权、交叉链接），所以漂移不可避免。

所以我们不把 AGENTS.md 当百科全书，而是当作**目录表**。

仓库的知识库存放在结构化的 `docs/` 目录中，作为记录系统。一个简短的 AGENTS.md（约 100 行）注入上下文，主要作为地图，指向其他更深层的真相来源。

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

设计文档被编目和索引，包括验证状态和一组定义 agent 优先运营原则的核心信念。架构文档提供领域和包分层的顶层地图。质量文档对每个产品领域和架构层评分，跟踪差距。

计划被视为一等公民。轻量的临时计划用于小改动，复杂工作则记录在执行计划（execution plans）中，含进度和决策日志，提交到仓库。活跃计划、已完成计划和已知技术债都被版本化并放在一起，允许 agent 无需依赖外部上下文即可运作。

这实现了**渐进式披露（progressive disclosure）**：agent 从一个小的、稳定的入口点开始，被教导下一步去哪里找，而不是一开始就被淹没。

我们用机械手段强制执行。专用 linter 和 CI 任务验证知识库是否最新、是否正确交叉链接和结构化。一个定期运行的「文档园丁」agent 扫描不反映真实代码行为的过时或废弃文档，并开修复 PR。

## Agent 可读性是目标

随着代码库演进，Codex 的设计决策框架也需要演进。

因为仓库完全由 agent 生成，它首先为 **Codex 的可读性**优化。就像团队致力于让代码对新工程师更易导航一样，我们人类工程师的目标是让 agent 能**直接从仓库本身**推理整个业务领域。

从 agent 的角度看，运行时上下文中无法访问的任何东西实际上都不存在。存在于 Google Docs、聊天记录或人脑中的知识对系统不可见。仓库本地的、版本化的制品（代码、markdown、schema、可执行计划）是它能看到的全部。

我们发现需要随时间推移将越来越多的上下文推入仓库。那个让团队在架构模式上对齐的 Slack 讨论？如果 agent 发现不了它，它就是不可读的——就像三个月后加入的新人不知道一样。

给 Codex 更多上下文意味着组织和暴露正确的信息，让 agent 能推理它，而不是用临时指令淹没它。就像你会向新队友介绍产品原则、工程规范和团队文化（包括 emoji 偏好）一样，给 agent 这些信息会带来更对齐的输出。

这个框架澄清了许多权衡。我们偏好能在仓库内完全内化和推理的依赖和抽象。通常被形容为「无聊」的技术往往更容易被 agent 建模，因为它们可组合、API 稳定、在训练数据中有充分表征。在某些情况下，让 agent 重新实现功能子集比处理公共库的不透明上游行为更划算。例如，我们没有引入泛型的 `p-limit` 风格的包，而是实现了自己的带并发限制的 map helper：它与我们的 OpenTelemetry 检测紧密集成，有 100% 测试覆盖率，行为完全符合我们运行时的预期。

将更多系统拉入 agent 能检查、验证和直接修改的形式，会增加杠杆——不仅对 Codex，对在代码库上工作的其他 agent（如 Aardvark）也是如此。

## 强制执行架构和品味

仅靠文档无法保持一个完全由 agent 生成的代码库的一致性。**通过强制执行不变量，而非微观管理实现方式，我们让 agent 快速交付而不破坏基础。** 例如，我们要求 Codex 在边界处解析数据形状，但不规定具体怎么做（模型似乎喜欢 Zod，但我们没指定这个特定库）。

Agent 在具有严格边界和可预测结构的环境中最为有效，所以我们围绕刚性架构模型构建应用。每个业务域被分为固定层次，严格验证依赖方向，只允许有限的边集合。这些约束通过自定义 linter（当然也是 Codex 生成的！）和结构化测试来机械执行。

规则是：在每个业务域内（如应用设置），代码只能沿固定层次「向前」依赖（Types → Config → Repo → Service → Runtime → UI）。横切关注点（认证、连接器、遥测、特性标志）通过单一显式接口进入：Providers。其他一切都被禁止且机械执行。

这种架构通常要等到你有几百名工程师时才会建立。有了编码 agent，它成为早期前提条件：约束才是允许速度而不衰退或架构漂移的关键。

实践中，我们用自定义 linter 和结构化测试加上一小组「品味不变量」来执行这些规则。例如，我们静态强制结构化日志、schema 和类型的命名约定、文件大小限制，以及平台特定的可靠性要求。因为 lint 是自定义的，我们在错误消息中注入修复指令到 agent 上下文。

在人类优先的工作流中，这些规则可能显得迂腐或限制性。在 agent 模式下，它们成为乘数：一旦编码，就在各处同时生效。

同时，我们明确哪里约束重要，哪里不重要。这类似于领导一个大型工程平台组织：中央强制边界，局部允许自主。你深切关心边界、正确性和可复现性。在这些边界内，你允许团队——或 agent——在解决方案的表达方式上有很大自由。

生成的代码不总是符合人类的风格偏好，这没关系。只要输出正确、可维护、对未来的 agent 运行可读，就达到了标准。

人类品味被持续反馈到系统中。Review 评论、重构 PR 和面向用户的 bug 被捕获为文档更新或直接编码到工具中。当文档不够时，我们将规则提升为代码。

## 吞吐量改变合并哲学

随着 Codex 吞吐量增加，许多传统工程规范变得适得其反。

仓库以最少的阻塞合并门运作。PR 生命周期很短。测试 flake 通常用后续运行解决，而非无限期阻塞进度。在一个 agent 吞吐量远超人类注意力的系统中，纠正成本低，等待成本高。

在低吞吐量环境中这样做是不负责任的。在这里，它往往是正确的权衡。

## 「Agent 生成」的真正含义

当我们说代码库由 Codex agent 生成时，我们指的是代码库中的一切。

Agent 产出：

- 产品代码和测试
- CI 配置和发布工具
- 内部开发工具
- 文档和设计历史
- 评估框架（evaluation harnesses）
- Review 评论和回复
- 管理仓库本身的脚本
- 生产仪表盘定义文件

人类始终在回路中，但工作在与以往不同的抽象层。我们确定工作优先级，将用户反馈转化为验收标准，验证结果。当 agent 遇到困难时，我们将其视为信号：识别缺少什么——工具、护栏、文档——并反馈到仓库中，始终由 Codex 自己来写修复。

Agent 直接使用我们的标准开发工具。它们拉取 review 反馈、内联回复、推送更新，经常自行 squash 并合并自己的 PR。

## 不断提升的自主级别

随着开发循环的更多部分被直接编码到系统中——测试、验证、review、反馈处理和恢复——仓库最近跨越了一个有意义的门槛：Codex 可以端到端驱动一个新功能。

给一个 prompt，agent 现在可以：

- 验证代码库的当前状态
- 复现一个报告的 bug
- 录制视频演示故障
- 实现修复
- 通过驱动应用来验证修复
- 录制第二个视频演示修复结果
- 开一个 PR
- 回应 agent 和人类的反馈
- 检测并修复构建失败
- 仅在需要判断时升级给人类
- 合并变更

这种行为严重依赖于这个仓库的特定结构和工具链，不应假设在没有类似投入的情况下能推广——至少目前还不行。

## 熵与垃圾回收

**完全 agent 自主也引入了新问题。** Codex 会复制仓库中已存在的模式——即使是不均匀或次优的模式。随着时间推移，这不可避免地导致漂移。

最初，人类手动处理。我们团队曾每周五（工作周的 20%）花时间清理「AI 泥浆」。不出所料，这无法扩展。

相反，我们开始将所谓的「黄金原则」直接编码到仓库中，并构建定期清理流程。这些原则是保持代码库对未来 agent 运行可读且一致的固执己见的机械规则。例如：（1）我们偏好共享工具包而非手写 helper，以保持不变量集中化；（2）我们不做 YOLO 式数据探测——我们在边界验证或依赖类型化的 SDK，这样 agent 不会意外地建立在猜测的形状之上。按固定节奏，一组后台 Codex 任务扫描偏差、更新质量评分、开有针对性的重构 PR。大多数可以在一分钟内 review 并自动合并。

这就像垃圾回收。技术债像高息贷款：持续小额偿还几乎总是比让它复利累积然后痛苦偿还更好。人类品味被捕获一次，然后在每一行代码上持续执行。这也让我们能每天发现和解决坏模式，而不是让它们在代码库中扩散数天或数周。

## 仍在学习的部分

这种策略到目前为止在 OpenAI 的内部发布和采用中运作良好。为真实用户构建真实产品帮助我们将投入锚定在现实中，引导我们走向长期可维护性。

我们尚不确定的是：在完全由 agent 生成的系统中，架构一致性如何随年份演进。我们仍在学习人类判断在哪里最有杠杆效应，以及如何编码这种判断使其持续复利。我们也不知道随着模型持续变得更强大，这个系统将如何演进。

已经清楚的是：构建软件仍然需要纪律，但纪律更多体现在脚手架而非代码本身。保持代码库一致的工具链、抽象和反馈循环越来越重要。

**我们现在最困难的挑战集中在设计环境、反馈循环和控制系统**，帮助 agent 实现我们的目标：大规模构建和维护复杂、可靠的软件。

随着 Codex 等 agent 承担软件生命周期的更大部分，这些问题将变得更加重要。我们希望分享一些早期经验能帮助你思考在哪里投入精力，这样你就可以直接去构建东西。

## My Comments

这篇文章是目前关于「全 Agent 开发」最有实操价值的分享之一。几个关键洞察：

1. **环境设计 > 提示工程**：工程师的核心工作不再是写代码，而是构建让 Agent 能高效工作的环境——工具链、知识结构、反馈循环。这比 prompt engineering 更深一层，更接近 platform engineering。

2. **「地图而非手册」**：一个大 AGENTS.md 不管用。需要结构化、可验证、有索引的知识库 + 渐进式披露。这其实也是人类团队的最佳实践，只是 Agent 让它变得更紧迫。我们自己的 CLAUDE.md 也应该遵循这个原则——保持简短，指向更深的文档。

3. **架构约束是加速器而非减速器**：在人类团队中可能显得 pedantic 的严格分层和 linting，在 Agent 开发中是必需品。没有约束，Agent 会快速产生架构漂移。自定义 lint 错误消息注入修复指令到 agent 上下文——这个技巧非常实用。

4. **技术债的「垃圾回收」机制**：让 Agent 自己清理 Agent 产生的债务，形成自我维护循环。将人类品味编码一次，然后持续自动执行。这比每周五手动清理「AI 泥浆」可持续得多。

5. **Agent 看不到的就不存在**：Slack 讨论、Google Docs、人脑中的知识——如果不在仓库里，对 Agent 来说就是不存在的。这迫使团队将所有重要上下文 push 到版本化的仓库制品中，反而形成了更好的知识管理。

6. **对个人的启示**：作为准备转向 AI 开发的工程师，「harness engineering」（驾驭工程）是一个重要概念。未来高价值工程师不是写最多代码的人，而是最擅长设计 Agent 工作环境的人——系统思维、架构品味、知识组织能力。这些恰好是资深工程师的强项。

**需要关注的风险**：文章没有讨论安全性（Agent 自动合并代码的风险）、成本（大量 Codex 调用的费用）、以及这种模式在非绿地项目（brownfield）中的适用性。这些都是真实世界中需要回答的问题。
