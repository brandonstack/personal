# Jeremy Howard：编程不是瓶颈，Claude Code 走错了方向

Jeremy Howard（fast.ai 创始人、ULMFiT 作者）与 Dr. Tim Scarfe 的播客对话。核心观点：AI 擅长编程但不擅长软件工程，Claude Code 式的开发方式正在削弱开发者能力。

原始视频：https://www.youtube.com/watch?v=dHBEQ-Ryo24

## 核心论点

### 编程 ≠ 软件工程

> 我大概有 90% 的代码都是由模型代劳，但这并没有显著提升效率，因为编程从来就不是效率的瓶颈。

Fred Brooks 几十年前在《没有银弹》中就预言过：技术进步最多带来 30% 的效率提升。软件工程的核心不是写代码，而是识别模块、设计组合、持续迭代——这需要几十年经验。

点名批评 Dario Amodei（Anthropic CEO）和马斯克：
- Dario 将顶尖工程师 + AI 的高效率粗暴推广到普通工程师，推断软件工程将被自动化——"这根本说不通"
- 马斯克声称 LLM 未来可直接输出机器码，不再需要库和编程语言
- **"这帮人都没当过现代软件工程师"**

### AI 的创造力是组合式的，无法突破训练分布

与 LLVM 创始人 Chris Lattner 讨论 Claude Code 写 C 编译器后的结论：
- LLVM 体系大量存在于训练数据中，转换为 Rust 本质是**风格迁移**
- 新项目直接复制了 LLVM 代码片段，包括 Chris 自认的设计错误
- "AI 是唯一照搬了 Chris 这种没人用的办法的开发者"

LLM 具备组合式创造力（重组训练数据中的已有知识），但无法突破训练分布边界。问题超出分布范围时，"模型突然就从极聪明变得特别蠢"。

### Claude Code 背离了人机交互的正确方向

人类软件史上最重要的创新（Smalltalk、APL、Mathematica）都强调人与计算机的紧密互动——实时操作对象、观察状态、调整参数。

> Claude Code 的运行环境跟人类 40 年前使用的环境极为相似——基于代码行的终端界面。你与系统的全部互动只是通过一个 prompt。这让我感到反感，有点不人道。

Jeremy 主张的方向：将人类与 AI 置于同一个交互环境（如 Python 解释器 / notebook），双方实时协作，共同构建工具。他的 SolveIt 产品就是这个思路。

### AI 编程正在侵蚀开发者能力

两类人真正受益于 AI 编程：
1. 完全不会编程的初学者——快速实现想法
2. 资深开发者（20+ 年经验）——让 AI 代劳部分编程和研究

**中间层（2-20 年经验）的大多数人是最危险的**——失去了通过亲手写代码来成长的机会。

> 如果团队成员停止成长，我们就注定会失败。

类比：学校禁止小学生用计算器来锻炼数感。学习需要"必要之难"（Ebbinghaus 间隔重复理论）——回避摩擦就学不到东西。Anthropic 自己的研究也发现大多数用 AI 编程的人无法展现学习曲线。

### 企业正在积累致命技术债

> 众多企业正因 AI 编程累积的技术债走向衰亡，这些债务使他们既无法维护现有产品、也难以开发新产品。

> 所以我觉得这就是在把企业和员工往被淘汰的绝路上推。

### 氛围编程像老虎机

> 伪装成胜利的失败、高度随机、虚假的控制感——这些都是博彩公司精心编排的元素。

METR 研究所发现人们进行 vibe coding 时生产力反而下降。Jeremy 表示几乎每位曾热情拥抱 AI 编程的人，回头审视成果后都改变了看法。

## 关于 ULMFiT 的历史背景

（播客前半部分回顾了 ULMFiT 的诞生过程：用 AWD-LSTM 在维基百科数据集上预训练，再微调→分类器。关键创新包括鉴别式学习率、逐层解冻、归一化层微调。这和大模型时代的"预训练+微调"范式一脉相承。）

## AI 风险

Jeremy 认为最大的现实风险不是 AI 变得太强大，而是：
1. **人类能力衰退**——丧失持续提升自身能力的机会
2. **权力集中**——强大技术集中在少数大公司和政府手中
