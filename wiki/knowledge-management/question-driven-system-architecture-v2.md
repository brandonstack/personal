# 问题导向思考系统 — 抽象架构 V2

> 从清明随笔的认知框架中提炼出的系统设计。两个视角描述同一个系统。

---

## 视角一：项目视角

### 系统是什么

一个**问题导向的思考系统**。用户通过提问、追问、分拆来推进对任何事情的理解和决策。

### 核心对象

| 对象 | 是什么 |
|---|---|
| **Project** | 一个有明确目标的事情。有一个 Deliverable（最终产出），这是项目存在的意义。 |
| **Question** | 待解决的问题。可以嵌套（大问题拆成小问题）。可以挂载思维模型作为拆分脚手架。 |
| **Evidence** | 支撑问题的事实和材料。 |
| **Thesis** | 你对某个问题的结论或理解。有了 Thesis，这个问题就算解决了。 |

### 一个项目长什么样

一棵树。根是 Deliverable（我要完成什么），枝干是 Question，叶子是 Evidence 和 Thesis。

```
Project (Deliverable)
  └── Question Tree
        ├── Q → 子Q → 子Q...
        │     ├── Evidence (支撑材料)
        │     └── Thesis (结论)
        └── Q → Thesis
```

### 节点状态

只有三种：
- ✅ 有 Thesis → 已解决
- ⚠️ 有 Question 但缺 Evidence 或 Thesis → 缺口
- ❓ 还没拆开 → 未展开

项目完成 = 所有关键 Question 都有 Thesis。缺口就是树上还没变绿的节点。

### 简单问题 vs 复杂问题

用的是**同一套结构**，区别只是树的深度：
- **简单问题**：一层就够。Q → 找点 E → 形成 T。完事。
- **复杂问题**：需要拆分。Q 拆成子 Q，子 Q 再拆，直到每个末端都能解决。可以挂载思维模型（SWOT、决策矩阵等）来辅助拆分。

### 问题树的生命周期

Question 提出 → 不断拆解追问 → 收集 Evidence → 形成 Thesis → 节点关闭。所有关键问题都有了 Thesis，项目就完成了。

整个项目就是一棵不断生长、不断变绿的树：
- 新 Question 冒出来 → 树长新枝
- Evidence 收集到 → 枝上挂果
- Thesis 形成 → 节点变绿
- 全部变绿 → Deliverable 可交付

## 视角二：知识视角

### 知识的三级

所有知识按消化程度分三级，统一管理，不区分来源：

| 层级 | 是什么 | 例子 |
|---|---|---|
| **Information** | 原始素材，未经加工 | 一篇文章、一个数据、一段对话、一条新闻 |
| **Knowledge** | 消化理解后的认知 | "重疾险在60岁以上性价比很低" |
| **Wisdom** | 抽象出的模型/框架/方法论 | "保险决策的核心变量是年龄×风险×预算" |

### 知识怎么来的

**Information 的来源：**
- 外部输入：搜索、阅读、视频、别人分享、RSS
- 现实反馈：数据变化、日志、市场价格
- 项目中收集的素材

**Information → Knowledge 的过程（消化）：**
- 对话、追问、发散、用自己的话讲出来
- 简单信息读一遍就懂，直接变 Knowledge
- 复杂信息需要拆解、对比、多轮对话才能理解
- **关键：这一步必须有人的参与，不能 AI 替你做完**
- Knowledge 本质是 Thesis 中沉淀下来的、可复用的部分

**Knowledge → Wisdom 的过程（抽象）：**
- 跨领域关联、抽象建模
- 当你在多个项目中反复遇到类似的 pattern，提炼出来就是 Wisdom
- Wisdom 不是一个项目就能产出的，是多个项目的 Knowledge 积累到一定量后的结晶

### 知识的流转循环

```
Information → (消化/对话/追问) → Knowledge → (跨领域关联/抽象) → Wisdom
     ↑                                                              │
     │                                                              ↓
     └──────────── 项目中的 Question 驱动收集 ←──── 作为思维模型指导新问题拆解
```

- 项目通过问题树消费知识
- 问题解决过程中产生新知识回流到知识池
- Wisdom 变成思维模型，回到问题树指导新项目的拆解
- **这就是系统的自我进化循环**

### 简单知识 vs 复杂知识

区别不在知识本身，在于消化的深度：
- **简单知识**：一条 Information 读一遍就懂，直接变 Knowledge
- **复杂知识**：需要多条 Information + 多轮对话 + 反复追问才能理解。消化过程就是不断拆解子 Question 的过程

**从知识的角度看，项目的问题树本身就是知识消化的过程图——树越深，消化越深。**

---

## 两个视角的统一

项目视角和知识视角描述的是**同一个系统的两面**：

- **项目视角**看到的是：问题树在生长、变绿、最终交付
- **知识视角**看到的是：Information 在被消化、提炼、进化

它们共享同一套数据结构（Q-E-T-D），同一套交互方式（提问、记录、总结），同一个原则（问题驱动、够用即可、知识即工具）。

---

## 核心原则

1. **问题驱动** — 一切从问题开始，不是从记录开始
2. **够用即可** — 不过度加工，服务目标就停
3. **一套模板** — 简单复杂都用 Q-E-T-D，项目和知识共用一套结构
4. **知识即工具** — 知识是拿来用的，不是拿来囤的
5. **人必须参与** — AI 辅助认知过程，但不替代
