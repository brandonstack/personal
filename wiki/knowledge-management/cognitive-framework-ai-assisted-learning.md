# AI辅助认知框架：从信息到智慧

> 清明随笔，2026-04-05。手写笔记整理。

## 起点：一个反思

最近一直被AI牵着走——整理inbox、build LLM-Agent体系、搞知识库，看起来很有成就感，但仔细一想都是AI写的，自己实际上什么也没沉淀，反而显得自己很渺小。

AI擅长文字工作，有无限token和精力。但人的注意力是有限的，一次无法处理过多事务，关注时长也有限。

每个人都有主观能动性——去努力、去克服、去实现。表达和思考，可能是人区别于AI的地方。

## 核心模型：四层认知阶梯

```
内容 → 信息 → 知识 → 智慧 → 应用
```

从原始内容出发：消化/理解成为知识，提炼抽象形成智慧，最终在实践中应用。人脑需要经历这个向上的过程，才算有所积淀。在与AI配合的今天，每一层AI应扮演不同角色。

### 第一层：Feed（内容→信息）— AI as 秘书

- AI负责收集、筛选、去重去噪
- 输出形式多样：简报、图片汇集、视频、数据可视化等
- **保留用户查看原始内容的能力**（可追溯）
- 广度：收集新进展、动态
- 深度：Research Agent负责某个主题的深挖（图表、原理解析）
- 目标：让人脑不需要接触噪音，只看精华——相当于秘书的角色
- 如果内容需要深入，也需要收集更多深入的内容。广度、深度的需求都在这里

### 第二层：Digest（信息→知识）— AI as 教练/老师

这是核心环节。**读一遍不算理解，读一遍放入知识库也不算。**

- 参考《如何阅读一本书》：讲解关键术语、关键语句、关联关系，分级消化
- AI的方式：对话、追问、发散、辅助理解、让用户讲出自己的理解
- **交互的效率仍然是最高的** — 不是给你一篇总结，而是跟你聊
- 语音聊天更好：可以发散，AI实时激发思考，打字容易变成"复制粘贴式学习"
- 延伸场景：AI备课、测试、盲问、考试 → AI As Teacher

### 第三层：Abstract（知识→智慧）— AI as 苏格拉底

- 抽象、建模——现实世界是不同模型的叠加
- 跨学科关联，找到不同知识之间的关键连接，实现真正的理解
- 知识库不应该有边界：可以是学习项目、写文章、关注某个领域
- 本质是一个**解构重组的知识库应用**

### 第四层：Apply（智慧→实践）— AI as 项目教练

光处理信息、阅读是不够的，不能排除行动和实践。

- **项目制**：把学到的东西拆成可执行的项目
- 每个项目需要：明确的最终目标、预定义的产出、阶段划分、顺序、规则及要求
- 一开始就需要定好产出是什么（买保险→真的去买；学技术→写代码产出）
- Project Knowledge + Project Dashboard 追踪进展
- 跨学科应用：在具体问题中整合不同领域的知识

## 关键洞察

1. **AI越强，人越容易变弱** — 如果AI替你做了所有认知工作，你自己没有成长
2. **AI的正确角色是辅助你走完认知过程，而不是替你跳过它**
3. **每一层需要不同的AI交互模式** — 不是一个chatbot打天下
4. **项目是学习的目的** — 没有方向的知识积累就是堆砌，项目驱动整个系统进化
5. **项目贯穿整个生命周期** — 很多需要知识性的东西，其实都是项目

## 产品形态思考：UI应该是什么样的？

### 以目标为入口

一开始便发问：**我们要做什么？** 有一个顶层主旨。从目标出发，所有内容才会更清晰。

以保险为例：
- 谁？→ 老人、年轻人
- 目的？→ 保障、养老
- 类型？→ 疾病、重疾、意外、养老保险
- 追问展开：产品选择、如何买保险、个人养老金…

思维导图式的提纲挈领，追问驱动层层展开。

### 够用即可，服务目标

从目标开始调整内容，删去无用功能。如果一些信息就够了，无需再往下挖。不用追求完美，够用即可。整理清楚，分门别类，必要时就不用深入。

**一切服务目标。**

### 存储极简

无需考虑复杂存储方式，纯文本即可。加上 Agent 就可以让它理解和处理。

### 自我进化

知识内容的自动整理也不可缺少——后台自动处理、重构（类似"睡眠整理"），让知识库自我进化。

## 项目驱动模型：Question → Evidence → Thesis → Deliverable

通过"买保险"和"学AI"两个例子的推演，形成了项目的核心四要素：

### 四要素

| 要素 | 含义 | 决策型（买保险） | 学习型（学AI） |
|---|---|---|---|
| **Question** | 待解决的问题 | 重疾还是医疗？ | attention怎么工作？ |
| **Evidence** | 事实 + 知识 | 保费对比、政策、年龄 | 论文、视频、代码实验 |
| **Thesis** | 你的结论/理解 | 买医疗不买重疾 | 我理解的attention是… |
| **Deliverable** | 最终产出 | 保险方案 + 购买完成 | 实现一个小模型 |

### 运转逻辑

**问题驱动**是核心。项目进展 = Question 不断转化为有 Evidence 支撑的 Thesis，循环往复，直到 Deliverable 完成。

系统状态不是"记了多少笔记"，而是"还有多少问题没解决"。

### Evidence 的两种归属

- **Project Knowledge**：跟项目强相关的事实（爸妈年龄、具体保费）——跟着项目走
- **General Knowledge**：跨项目可复用的知识（保险行业运作方式、attention原理）——沉淀为知识库

### Thesis 的两种性质

- **决策型**：终态，决定了就执行（买医疗险）
- **学习型**：当前最佳理解，可迭代推翻（对attention的理解随深入学习更新）

### 项目交互流程推演（以保险为例）

```
用户："我想买保险"
  → AI追问澄清：给谁？为什么？（养老？风险保障？）
  → 收集用户信息：年龄、身体状况、现有保障
  → 生成 Plan：关键问题列表 + 知识缺口
  → Research Agent 补知识（Evidence）
  → 用户 + AI 对话消化，形成 Thesis
  → 所有关键问题解决 → Deliverable 交付
```

### 问题的三种来源

- **AI能做的问题** → 自动研究（Research Agent）
- **推动项目的问题** → 需要人来决策
- **人问自己的问题** → 反思、价值判断

问题会在知识获取过程中不断涌现（知识越多，新问题越多），但项目整体有一个宏观的决策树/状态图，确保方向不迷失。

## 统一项目图：一棵不断变绿的树

项目图和缺口分析不是两个功能，而是**同一张图的不同视角**。

### 一棵树，三种节点状态

```
买保险 (Deliverable: 完成方案)
├── Q: 买什么险种？ ✅ 已解决
│   ├── E: 爸60岁，重疾保费高
│   ├── E: 医疗险性价比更优
│   └── T: 先买百万医疗
├── Q: 哪家产品好？ ⚠️ 缺口
│   ├── E: (需要对比3家)  ← 缺Evidence
│   └── T: (待决策)
└── Q: 预算多少？ ❓ 未展开
```

- ✅ **已解决** = 有 Thesis（论点已成）
- ⚠️ **缺口** = Question 已明确，但 Evidence 不够或 Thesis 未形成
- ❓ **未展开** = Question 还没 break down

**缺口就是树上还没变绿的节点。** 不需要单独的缺口分析视图——折叠看全局，展开看细节，一张图搞定。

### Question 的双重身份

Question 既是**驱动力**（推动你去研究），也是**缺口的表达形式**（发现缺口 = 发现新 Question）。用 Question 来逐步 break down，直到每个末端节点都有 Thesis。

### 项目的生命周期

整个项目就是一棵不断生长、不断变绿的树：
- 新 Question 冒出来 → 树长新枝
- Evidence 收集到 → 枝上挂果
- Thesis 形成 → 节点变绿
- 全部变绿 → Deliverable 可交付

### 自动化能力

基于这棵树，系统可以自动：
- 构建并更新项目图（思维导图式展示）
- 计算全局状态（活跃项目数、未解决问题数、已形成论点数）
- 识别缺口并推荐下一步行动（哪些 Question 最需要 Evidence？）
- 从 Deliverable 反推完成度

## 为什么这个框架有效？

### 不复杂

核心就四个概念（Q-E-T-D），用户日常操作就三件事：提问题、记证据、写结论。树是自动长出来的。比 Notion 的 database/relation/rollup 简单，比 Obsidian 的 tag/link/graph 更有方向感。

### 砍掉任何一个都出问题

- 没有 Q → 不知道要干嘛
- 没有 E → 拍脑袋决策
- 没有 T → 永远在收集不做决定
- 没有 D → 学了一堆没有产出

### 视角翻转

之前的 Inquiry 版本都从工具出发——怎么做 inbox、怎么做 digest、怎么存 atom。这次从**认知过程**出发——人怎么学习、怎么做决策——Q-E-T-D 自然浮现。从工具思维到认知思维，是这次突破的关键。

## 框架扩展：不只是知识管理

### Q-E-T-D 是通用的项目推进框架

不只适用于学习和决策项目，数据驱动的现实项目同样适用：

- **工作项目**（如提高 price accuracy）：Q=为什么准确率低？ E=数据分析结果 T=断掉 organic 源 D=指标提升
- **投资项目**（如股票）：Q=该不该买？ E=财报、市场数据、实时反馈 T=买入/观望 D=投资决策执行
- **软件项目**：Q=用户为什么流失？ E=日志、用户反馈、A/B测试 T=改进方案 D=功能上线

> **知识即为工具，而非是本身。**

### Evidence 的分级处理

按信息→知识→智慧的层级，Evidence 不需要统一深加工：

- 有些停在**信息层**即可（剪藏、日志、原始数据）
- 有些需要提炼为**知识**（分析报告、对比总结）
- 有些需要上升到**智慧**（建模、框架、方法论）

够用即可，不过度加工。服务目标。

### Evidence 的来源

- 主动搜索（Research Agent）
- 项目相关的 clipping/剪藏
- 现实反馈（股价、价格变动、错误日志、社区讨论）
- 对话消化中产生的理解

## 复杂项目：模型嵌套

Q-E-T-D 天然支持嵌套——大 Question 拆成小 Question，树长得更深。

更强大的是：可以在 Question 节点上**挂载思维模型**作为分拆的脚手架。

```
Q: 要不要离职做独立开发？
  ├── [SWOT模型]
  │   ├── Q: Strengths - 我有什么优势？
  │   │   ├── E: 大厂后端经验
  │   │   ├── E: AI/Agent 深度理解
  │   │   └── T: 技术+产品直觉是核心优势
  │   ├── Q: Weaknesses - 短板？
  │   ├── Q: Opportunities - 市场机会？
  │   └── Q: Threats - 风险？
  └── T: (综合SWOT后的决策)
```

- 不同 Question 可以用不同模型（SWOT、决策矩阵、费曼学习法、第一性原理…）
- 模型是**思考的脚手架**，让分拆有结构而非漫无目的
- 可以 leverage 已有模型，不用全部自己发明

### 从树到思考方式的外化

这就不只是一棵任务树了——**树的形状就是你脑子里的思路**。

- 可以渲染为思维导图
- 可以变成甘特图（加上时间维度）
- 可以导出成报告
- 本质上是同一棵树的不同视图

### 用即学，学即用

用模型去拆解 Question → 拆解过程中学会这个模型 → 模型变成 General Knowledge → 以后别的项目也能用。

知识在使用中积累，而不是在阅读中囤积。

## 思维外化：通用分析工具

### UI形态

类似 Roam Research 的 bullet point 嵌套，作为思考空间。Thinking tools（SWOT、金字塔原理等）可以作为模板直接嵌入到 Question 节点中，成为遍历式分析工具。

### 知识消化即项目

对复杂知识的消化本身就是一个"进行中的项目"——用 Q-E-T-D 走一遍分析流程，产出就是你的理解。简单内容直接归类，复杂信息则在实际运用后才有深度理解。

### 跨学科关联

AI 可以发起相似度审核（"这个概念跟你另一个项目里的X有关联"），然后人继续追问、判断。

### 定位：问题导向的思考操作系统

Q-E-T-D + 模型嵌套不是某个特定场景的工具，是一个**通用的分析模型**（General Analysis Model）。买保险用它，学AI用它，消化论文用它，做工作项目还是用它。

#### 学术验证

这个方向有成熟的学术根基：

- **Problem-Based Learning (PBL)**：哈佛医学院1985年开始使用。核心：不先教知识，先给问题，围绕问题找证据、形成理解。Q-E-T-D 与 PBL 底层逻辑一致，加了 Deliverable 和树状可视化。
- **Issue-Based Information System (IBIS)**：1970年代的复杂问题分析方法，核心结构 Question → Position → Argument，与 Q → T → E 本质相同。后发展出 Compendium 工具。

不是发明全新的东西，而是**重新发现一个被验证过的认知模式，加上AI时代的新能力**。

#### 为什么适合"刨根问底"

系统的核心动作是**提问**，而非记录：
- 记录是被动的：看到什么记什么
- 提问是主动的：不懂什么问什么

提问是人类理解复杂系统最有效的方式。AI的加持让它更强——帮你发现 unknown unknowns，帮你拆解问题，帮你找证据。人负责判断和决策。

#### 产品化挑战

PBL 和 IBIS 有几十年历史但未成为主流工具——因为大部分人不喜欢被追问、不喜欢暴露不懂的东西。关键在于：**怎么让"提问"变得轻松而非有压力**。AI在这里可以起很大作用——它来问你，而不是你来填表。
