# AI Infra & Deployment — Knowledge Sprint 消化报告

## 这批材料在说什么

这批 5 篇有效材料（Maarten Grootendorst 和 NVIDIA 的页面 404/无内容）提供了 LLM 推理优化的完整图景：从 GPU 硬件特性（Coding Confessions）到推理成本分析（Cursor）到 serving 优化（Anyscale continuous batching）到工具生态（vLLM docs）到学术综述（Wan et al.）。

Coding Confessions 的 GPU 文章和 Cursor 的推理成本分析信息密度最高，Anyscale 的 continuous batching 文章提供了 23× 吞吐提升的关键 benchmark。

## 关键洞察

1. **LLM 推理的核心矛盾是 "计算多但空闲多"**：GPU 有 8448 个 CUDA cores（H100），但 decode 阶段每步只做一次矩阵-向量乘法，大部分时间在等 HBM 读取数据。Memory bandwidth（3 TB/s for H100）比 compute（67 TFLOPS FP32）更决定推理性能。这个 insight 颠覆了"LLM 推理就是大量计算"的直觉。

2. **Generation 比 Prompt Processing 贵 ~160×（batch=1）**：$0.066 vs $0.00042 per 1K tokens（Llama-2-70B on 2×A100）。这解释了所有 API 定价策略和 output token 限制的经济学基础。也解释了为什么 "thinking tokens"（如 Claude 的 extended thinking）成本很高——它们都是 generation tokens。

3. **PagedAttention 是 OS 虚拟内存概念在 AI 系统中的优美复用**：传统 KV Cache 的 "预分配连续内存 + 碎片浪费" 和 1960s 操作系统的内存管理问题一模一样。vLLM 的 page table + 非连续分配 + 按需释放直接复用了 OS 的解法。这种跨领域的概念迁移是 AI Systems Engineer 的核心竞争力。

4. **Batch size 是推理经济学的核心杠杆**：batch=1 时 generation 成本 $0.066/1K tokens，batch=32 时降到 $0.0039/1K tokens（17× 降低）。本质是多个请求分摊同一次权重读取的内存带宽成本。这也是 continuous batching 如此重要的原因——它最大化了有效 batch size。

5. **Quantization 的 "2× 规则"**：INT8 → 内存减半、速度约 2×、质量几乎无损。这是推理部署的最低 hanging fruit。更激进的 INT4 也可用，但需要 eval 验证。

## 与已有知识的关系

- **KV Cache 跨越三个 wiki 主题**：ml-fundamentals/kv-caching.md（原理）、ai-infra/continuous-batching.md（PagedAttention 优化）、ai-infra/llm-inference-cost.md（成本影响）。这是 wiki 中最强的跨主题连接之一。
- **GPU 的 memory hierarchy 和 claude-code 的 context 管理是同一问题的不同层面**：Registers > Shared Memory > L2 > HBM 对应 system prompt > recent messages > retrievable context > full history。两者都是在"有限容量的多层存储"中做最优分配。
- **Continuous batching 的 "iteration-level scheduling" 和 harness 的 "real-time state management" 有架构相似性**：两者都是在动态环境中做细粒度调度。

## 对你的具体建议

1. **GPU Architecture 是你作为后端工程师最容易建立可信度的领域**。你懂 OS 概念（虚拟内存、page table、调度）——PagedAttention 就是你擅长的东西在 AI 领域的应用。面试中讲 "我理解 LLM 推理为什么慢，以及怎么用系统工程的方法让它快" 远比 "我会用 vLLM" 有说服力。

2. **建议在微软服务器上跑一次 vLLM serving 实验**：用开源模型（Llama-2-7B 就够），实测 static batching vs continuous batching 的吞吐差异。不需要生产级——关键是获得第一手数据和直觉。这可以作为项目 2 的 infra 组件。

3. **Cursor 的成本分析文章可以作为你理解 API 定价的框架**。当你在项目中选择 self-host vs API 时，可以用文中的公式估算成本。这种"工程经济学"分析能力是 AI Systems Engineer 的加分项。

4. **Wan et al. 的综述论文值得深入读 PDF 版**（这次只抓到了 arxiv 摘要）。它的分类框架（data-level, model-level, system-level 优化）可以作为你 infra 知识的组织框架。

## 值得讨论的问题

1. **你对 infra 的定位深度是什么？** "了解 GPU 架构和推理优化原理" vs "能部署和调优 vLLM serving"——前者是知识，后者是技能。对 AI Systems Engineer 来说，哪个更重要？还是两者都需要？

2. **自托管 vs API 的决策框架**：你的项目中会使用 Claude API（对齐 Anthropic 生态）还是自托管开源模型（展示 infra 能力）？考虑到你的定位叙事是 "AI Systems Engineer"，是否应该两者都做（项目 1 用 API，项目 2 用 self-hosted）？

3. **KV Cache 的 memory management 和 OS virtual memory 的类比可以写成一篇博客**：这是一个非常有话题性的角度——"为什么 1960s 的操作系统概念能解决 2024 年的 AI 问题"。对你的"后端工程师视角"博客系列来说，这可能是最容易写也最有传播力的一篇。
