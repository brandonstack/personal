# Agent 环境设计：从写代码到设计 Agent 工作环境

OpenAI Codex 团队的核心发现：工程师的角色从写代码转变为设计让 Agent 高效工作的环境。5 个月用 Agent 构建约 100 万行生产代码，零行手写。

## 环境设计 > 提示工程

早期进展比预期慢，不是 Agent 能力不够，而是环境描述不充分。Agent 缺少向高层目标推进所需的工具、抽象和内部结构。

当某件事失败时，修复方式不是"再试一次"，而是问："缺少什么能力？如何让它对 Agent 既可见又可执行？"

人类通过三件事驱动开发：设计开发环境、用结构化 prompt 表达意图、给 Agent 提供反馈循环。

## 知识组织：地图而非手册

"一个大 AGENTS.md"的方式可预见地失败了：

- **Context 是稀缺资源**——巨大指令文件挤占任务和代码的空间
- **太多指导 = 非指导**——当一切都是"重要的"，就没有什么重要
- **立刻腐烂**——单体手册变成陈旧规则的坟场
- **难以验证**——单一大块不适合机械检查

解法：AGENTS.md 当目录表（~100 行），指向结构化的 `docs/` 目录。实现**渐进式披露（progressive disclosure）**：Agent 从小的、稳定的入口点开始，被教导下一步去哪里找。

知识库包含：设计文档（含验证状态和核心信念）、执行计划（进度和决策日志）、质量评分、架构地图。全部版本化，Agent 无需依赖外部上下文即可运作。

## Agent 可读性原则

从 Agent 的角度看，运行时上下文中无法访问的任何东西实际上都不存在。Slack 讨论、Google Docs、人脑中的知识——对 Agent 不可见就是不存在。

这迫使团队将所有重要上下文 push 到版本化的仓库制品中。偏好"无聊"技术——可组合、API 稳定、训练数据中有充分表征。有时让 Agent 重新实现功能子集比处理公共库的不透明行为更划算。

## 架构约束是加速器

不变量通过 linter + CI 机械执行，而非文档自觉遵守：

- 依赖层级严格分六层：Types → Config → Repo → Service → Runtime → UI
- 横切关注点通过单一显式接口（Providers）进入
- 违反约束的代码自动被拒

Linter 本身也是 Agent 生成的。错误消息中注入修复指令到 Agent 上下文——这个技巧使 Agent 能自动修复违规。

> "增加信任和可靠性，需要约束解空间。这意味着放弃一些'生成任何东西'的灵活性。" — Martin Fowler

约束的哲学：中央强制边界，局部允许自主。深切关心边界、正确性、可复现性。在边界内，允许 Agent 在解决方案的表达方式上有很大自由。

## 熵对抗与垃圾回收

Agent 系统跑久了会自然腐化。解法不是人工清理（无法扩展），而是：

- 将"黄金原则"编码到仓库中
- 定期后台 Agent 扫描偏差、更新质量评分、开重构 PR
- 大多数可以一分钟内 review 并自动合并

技术债的垃圾回收机制：人类品味被捕获一次，然后在每一行代码上持续自动执行。

> "技术债像高息贷款：持续小额偿还几乎总是比让它复利累积然后痛苦偿还更好。"

## 吞吐量改变合并哲学

Agent 吞吐量远超人类注意力 → 纠正成本低、等待成本高 → PR 快速合并、test flake 用后续运行解决。

在低吞吐量环境中这样做不负责任。在 Agent 驱动的高吞吐量环境中，这往往是正确的权衡。
